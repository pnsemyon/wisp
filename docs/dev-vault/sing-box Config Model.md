# sing-box Config Model

#singbox #config

This note walks through the JSON config that [[Crate - wisp-core]]'s `build_config()` produces
— the same JSON that [[Crate - wisp-engine]]'s `SingBoxProcess` writes to `config.json` and
hands to `sing-box run -c config.json`. If you haven't yet, skim [[Glossary]] for terms like
[[Glossary#REALITY|REALITY]], [[Glossary#TUN|TUN]], and [[Glossary#selector|selector]] used
below.

## Annotated example

Given the 3-outbound fixture profile used throughout `wisp-core`'s tests (two VLESS+REALITY
outbounds, one Hysteria2), `off` split mode, and default `BuildSettings`, `build_config`
produces (abbreviated, comments added):

```jsonc
{
  "log": { "level": "info", "timestamp": true },

  "dns": {
    "servers": [
      // remote/proxied resolver — routed through the tunnel so DNS itself
      // doesn't leak to the local ISP
      { "tag": "dns-remote", "address": "tls://8.8.8.8", "detour": "proxy" },
      // local resolver — used for anything routed direct
      { "tag": "dns-local", "address": "local", "detour": "direct" }
    ],
    "strategy": "prefer_ipv4"
  },

  "inbounds": [
    {
      // the virtual network adapter that captures ALL system traffic
      "type": "tun",
      "tag": "tun-in",
      "mtu": 1280,                       // from BuildSettings.mtu, see [[Glossary#MTU|MTU]]
      "address": ["172.19.0.1/30"],
      "auto_route": true,                // sing-box manages OS routing table itself
      "strict_route": true,              // route ALL traffic through the TUN, no leaks
      "stack": "system",                 // use the OS network stack (vs. gVisor/mixed)
      "endpoint_independent_nat": false
    }
    // + an optional "socks" inbound on 127.0.0.1:<socks_port> if configured
  ],

  "outbounds": [
    // --- the profile's own outbounds, included VERBATIM ---
    { "type": "vless", "tag": "Bulgaria, Sophia-7w1t0rtt5a § 0", "...": "REALITY+XHTTP" },
    { "type": "vless", "tag": "Bulgaria, Sophia-7w1t0rtt5a § 1", "flow": "xtls-rprx-vision", "...": "REALITY" },
    { "type": "hysteria2", "tag": "Bulgaria, Sophia, hysteria-7w1t0rtt5a § 2", "...": "" },

    // --- generated by build_config ---
    {
      "type": "selector",
      "tag": "proxy",
      "outbounds": [
        "Bulgaria, Sophia-7w1t0rtt5a § 0",
        "Bulgaria, Sophia-7w1t0rtt5a § 1",
        "Bulgaria, Sophia, hysteria-7w1t0rtt5a § 2"
      ],
      "default": "Bulgaria, Sophia-7w1t0rtt5a § 0"   // profile.active_tag, or first tag
    },
    { "type": "direct", "tag": "direct" },
    { "type": "block", "tag": "block" }
  ],

  "route": {
    "auto_detect_interface": true,
    "final": "proxy",              // where unmatched traffic goes (see [[Split Tunneling]])
    "rules": [
      { "action": "sniff" },
      { "protocol": "dns", "action": "hijack-dns" },
      { "ip_is_private": true, "outbound": "direct" }
      // ...split-tunnel rules inserted here when mode != Off, see [[Split Tunneling]]
    ]
  },

  "experimental": {
    "clash_api": {
      "external_controller": "127.0.0.1:9090",   // BuildSettings.clash_port
      "secret": ""                                // BuildSettings.clash_secret
    }
  }
}
```

## How the user's 3 outbounds map in

`profile.outbounds` is copied into `config["outbounds"]` **unchanged** — `build_config` never
edits an outbound's fields. It only *appends* to the outbounds array:

1. A `selector`-type outbound tagged `"proxy"`, whose `outbounds` list is exactly
   `profile.tags()` (every outbound's `"tag"`, in order). Its `"default"` is
   `profile.active_tag` if that tag still exists among the current outbounds, else the first
   tag. This selector is what route rules actually point traffic *at* — not the individual
   VLESS/Hysteria2 outbounds directly — so switching servers at runtime is just a
   `PUT /proxies/proxy` Clash API call (see [[Crate - wisp-engine]]) rather than regenerating
   the whole config.
2. A `direct` outbound (`tag: "direct"`) — traffic bypasses the tunnel entirely.
3. A `block` outbound (`tag: "block"`) — present for completeness/future use; nothing in
   `build_config` currently routes to it by default.

## `inbounds`: the `tun` entry

The `tun` inbound is what makes this a *system-wide* VPN rather than a per-app SOCKS proxy: with
`auto_route: true` and `strict_route: true`, sing-box modifies the OS routing table so all
traffic (subject to `route.rules`) flows through the virtual adapter. `mtu` is written directly
here from `BuildSettings.mtu` (default 1280) — this is Wisp's answer to "no more manual MTU
fiddling" from the project's README: no post-launch `netsh`/`ip link set mtu` commands, it's
just a config field. `stack: "system"` uses the OS's own TCP/IP stack (as opposed to sing-box's
userspace/gVisor stack option).

An optional second inbound (`socks`, listening on `127.0.0.1:<socks_port>`) is added only if
`BuildSettings.socks_port` is `Some` — currently unused by the Tauri backend (`socks_port` is
always `None` in `commands::connect`), but available to CLI users via `wisp-cli` (not currently
exposed as a flag) or future use.

## `dns`

Two resolvers: a remote DoT (`tls://8.8.8.8`) resolver detoured through `proxy` (so DNS queries
for tunneled destinations don't leak locally), and a `local` resolver detoured `direct` (for
locally-resolved names). `strategy: "prefer_ipv4"` avoids IPv6-related connectivity issues.

## `route`

See [[Split Tunneling]] for the full rules logic — this note covers only the *always-present*
rules and overall shape. Rules are evaluated top-to-bottom, first match wins:

1. `{"action": "sniff"}` — sniff the protocol/domain of each connection (needed for
   `domain_suffix` rules to have something to match against).
2. `{"protocol": "dns", "action": "hijack-dns"}` — intercept DNS queries so they're answered by
   the `dns` block above rather than escaping the tunnel.
3. `{"ip_is_private": true, "outbound": "direct"}` — always send RFC1918/local traffic direct,
   regardless of split-tunnel mode, so LAN access (printers, routers, etc.) never breaks.
4. *(split-tunnel rules, if any — see [[Split Tunneling]])*

`route.final` is the default outbound for anything that falls through every rule — `"proxy"`
unless `Include` mode sets it to `"direct"`.

## `experimental.clash_api`

`external_controller` binds sing-box's Clash-compatible HTTP API to
`127.0.0.1:<BuildSettings.clash_port>` (default 9090), guarded by `secret`. This is the *only*
channel [[Crate - wisp-engine|wisp-engine]]'s `ClashApi` uses to query traffic totals
(`/connections`) and switch the active server (`PUT /proxies/proxy`) — see
[[Crate - wisp-engine]].
